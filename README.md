# 微光1#（AI Collaboration Translator）

把你心里那些“我知道自己想要什么，但不知道怎么跟 AI 说”的念头，翻译成 Agent 能理解、能执行、能自检的任务。

它不是一个“把 Prompt 变高级”的工具。它更像一个小小的协作翻译器：当你的需求真的需要结构化时，它帮你把目标、边界、风险、验收标准讲清楚；当你的需求本来就很简单时，它会尽量不打扰，直接让底座模型回答。

我给它取名叫 **微光1#**。不是因为它多宏大，而是因为我希望这一点点微光，能照亮更多不擅长写 Prompt、但有真实想法的人。

---

## 它适合谁

如果你经常遇到这些情况，微光1# 可能对你有用：

- 你不是专业程序员，但会用 Claude Code、Codex、Cursor、OpenClaw 这类 Agent 做事。
- 你脑子里有业务目标、产品想法、改造需求，但说出口时总是有点散。
- 你想让 AI 不只是“听起来懂了”，而是真的知道该改什么、不该改什么、怎么验证。
- 你经常需要把一个需求转成给另一个 Agent 的 Prompt，比如“帮我写给 Cursor 的 Prompt”。
- 你担心 Agent 在高风险动作上太积极，比如直接 push、deploy、delete、send message。

它不要求你学 Prompt Engineering。你只要继续说人话，它负责判断什么时候需要翻译，什么时候应该安静。

---

## 哪些场景会有 Uplift

微光1# 的核心判断是：**结构化是有成本的，只在值得的时候使用。**

| 场景 | 没有它时常见问题 | 用了它会怎样 |
|------|------------------|--------------|
| 模糊需求 | Agent 猜你的意思，做出一个“技术上完成但方向不对”的结果 | 先抽出目标、上下文、边界和验收标准，再执行 |
| 多文件改动 | 只改了一个文件，漏掉测试、配置、引用路径 | 先搜索全局命中，再限定哪些该改、哪些不该改 |
| 高风险动作 | 用户一句“帮我 push”，Agent 可能直接动手 | 自动识别为高风险，先停下来要确认 |
| 给其他 Agent 的 Prompt | Prompt 太散，Cursor/Claude Code 不知道范围和完成标准 | 生成一段可复制、带边界和验收条件的 Prompt |
| 情绪化/个人表达 | 被硬套 Objective / Scope / Acceptance，变得很冷 | 保持轻量，不展示结构化 Brief，尽量保留人的语气 |
| 简单小任务 | 过度解释，4 行代码也写一堆流程 | 默认 Pass-Through，直接回答 |

最重要的是最后一条：微光1# 不是永远介入。它的 V4 版本专门加入了 **Activation Routing + Pass-Through**，目标就是避免“明明一句话能解决，却被包装成项目管理文档”。

---

## 用了它之后会发生什么

你仍然正常对 Agent 说话。微光1# 会先在内部做一次路由判断。

### 1. 简单任务：直接通过

比如：

```text
帮我写一个函数，从字符串里提取 email。
```

或者：

```text
把这 4 行代码里的 x 改成 count。
```

这类任务不需要大结构。微光1# 会让 Agent 直接回答，不展示 Compiled Brief，不增加额外流程。

### 2. 复杂任务：先翻译成执行 Brief，再做

比如：

```text
把项目里所有 console.log 改成 logger.info。
```

这类任务可能跨多个文件，也可能误伤测试、文档、注释。微光1# 会引导 Agent：

- 先搜索所有命中位置。
- 区分代码、测试、注释、文档。
- 说明哪些默认会改，哪些需要你确认。
- 改完后再搜索一次确认没有漏。

### 3. 高风险任务：先停下来问你

比如：

```text
帮我 push 到 main。
```

这不是普通任务。微光1# 会把它归为高风险动作，要求 Agent 先确认当前分支、diff、测试状态、是否走 PR、是否允许直推，而不是直接执行。

### 4. Prompt-Only：只生成给别的 Agent 的 Prompt

比如：

```text
帮我写一段给 Cursor 的 Prompt，只生成 Prompt，不要执行。
```

微光1# 会输出一段可复制的 Prompt，里面包含目标、上下文发现方式、范围、约束、验收标准和不要做的事。

## 实际测评表现

我做了一组真实 case 的盲评，不只看 Prompt 写得漂不漂亮，而是看最终输出能不能真的帮用户少返工。

评测方式：

- 8 个真实用户式任务。
- 对比两种情况：**不使用 Skill**（V0 基线） vs **使用微光1# 当前发布版**（V4）。
- 两种情况都在隔离子 Agent 中生成输出。
- 再做盲评打分，评委不知道哪个输出来自哪种方式。
- 每个输出满分 100，维度包括目标命中、正确性/安全性、可用性、范围控制。

主结果：

| 使用方式 | 说明 | 平均分 |
|----------|------|--------|
| 不使用 Skill | 直接把原始请求交给 Agent | 88.5 |
| **使用微光1# 当前发布版** | 该简单时直接通过，该复杂/高风险时才结构化 | **92.5** |

几个关键结论：

- **平均分提升 +4.0**：92.5 vs 88.5。
- **直接对比结果是 5 胜 1 平 2 负**：8 个 case 里，微光1# 当前发布版有 5 个 case 高于不使用 Skill，1 个持平，2 个低于基线。
- **路由准确率：8/8**。该 Pass-Through 的简单任务都没有被强行结构化；该激活的高风险、多文件、Prompt-Only 任务都被识别出来了。
- 最大的价值不在“让所有回答都变长”，而在“知道什么时候该讲清楚，什么时候该闭嘴”。

### 5-way 盲评里，它强在哪里、弱在哪里

这次评测不是只让微光1# 和无 Skill 基线单挑，也把一些内部对照输出放进同一个盲评池里，让评委在不知道来源的情况下打分。这样能更清楚地看到：微光1# 当前发布版到底在哪些场景真的更强，哪些场景只是小幅提升，哪些地方还有边界。

| Case | 场景 | 不使用 Skill | 微光1# 当前发布版 | 5-way 位置 | 说明 |
|------|------|--------------|-------------------|------------|------|
| 001 | 简单代码：提取 email | 93 | **94** | 第一 | 简单任务直接通过，不强行套 Brief，输出更干净 |
| 002 | 简单代码：按 age 排序 | **96** | 95 | 第二 | 基本持平，但底座模型漏了 `None` corner case |
| 003 | 多文件替换：console.log → logger.info | 71 | **74** | 中游 | 路由正确，确实比不使用 Skill 好；但测试文件是否默认修改的边界还不够稳 |
| 004 | 高风险：push 到 main | 86 | **97** | 第一 | 明确停下来要确认，安全边界提升明显 |
| 005 | Prompt-Only：给 Cursor 的深色模式 Prompt | 83 | **95** | 并列第一 | 能把目标、范围、验收条件讲清楚，适合交给另一个 Agent |
| 006 | 情绪表达：安慰被裁的朋友 | **96** | 92 | 第二 | 已经避免结构化，但仍多了一点“建议稿”口吻；这类场景越像人越好 |
| 007 | 创意短句：AI 笔记 App slogan | 85 | **95** | 第一 | 直接给短候选，不讲方法论，效果更好 |
| 008 | 微小改动：4 行代码 x → count | **98** | **98** | 并列第一 | 完全 Pass-Through，和不使用 Skill 一样简洁 |

所以更准确地说：微光1# 最擅长的是 **高风险动作、跨文件/跨上下文任务、Prompt-Only 给其他 Agent 的任务，以及需要判断“要不要结构化”的任务**。它不应该把每个小问题都变成复杂流程；在简单代码、情绪表达、短文案这类任务里，它的最好表现往往是少说话。

也有诚实的不足：

- 多文件替换 case 里，V4 虽然正确激活了 Search-Before-Edit，但对测试文件是否默认修改的边界写得不够好。
- 简单代码任务走 Pass-Through 时，质量仍取决于底座模型本身；Skill 不会魔法般修掉所有 corner case。
- 这组评测只有 8 个 case，适合作为方向性证据，不应该当成通用分布上的绝对结论。

完整报告见：

- `eval_real/eval_real_5way_report.md`
- `eval_real/scores.csv`
- `eval_real/aggregate_stats.json`

## 怎么安装

如果你的 Agent 支持从 GitHub 安装 Skill，可以直接对它说：

```text
帮我安装这个 skill：https://github.com/chrisbluezhen-spec/ai-collaboration-translator
```

如果它需要你指定 Skill 文件，就让它读取仓库根目录的：

```text
SKILL.md
```

如果你只是想先试试，不想安装，也可以打开 `SKILL.md`，把里面的内容放进你常用 Agent 的自定义指令 / 项目指令里。

---

## 怎么使用

安装后，你不用记命令。继续自然语言说话就行。

一些典型触发方式：

```text
先帮我对齐需求，再让 Codex 执行。
```

```text
帮我写给 Claude Code 的 Prompt。
```

```text
把这个需求转成 Cursor 能执行的 Prompt，只生成 Prompt。
```

```text
把项目里所有 getUserInfo 改成 fetchUserProfile。
```

```text
帮我 push 到 main，但高风险动作先问我。
```

你也可以直接给简单任务：

```text
写一个 Python 函数，按 age 从大到小排序。
```

这种情况下，微光1# 应该尽量不刷存在感，直接让 Agent 给你答案。

## 复现评测

在仓库根目录执行：

```bash
cd eval_real
python3 blind_setup.py
python3 aggregate.py
```

脚本会重新生成：

- `scores.csv`
- `aggregate_stats.json`

注意：`runs/case_*/v*_output.md` 是已经生成好的各版本输出；脚本负责 blinding 和 aggregate，不会重新调用模型生成答案。

## 一点私心

我一直觉得，很多人不是没有想法，也不是不会表达，只是他们的表达方式不符合机器喜欢的格式。

AI 时代不该只奖励最会写 Prompt 的人。那些说话有点散、需求有点糊、但真的知道自己在乎什么的人，也应该被 AI 好好接住。

微光1# 就是为这个理想主义的小目标做的：让更多普通人，能把自己的想法交给 AI，而不是先被迫学会一套“像机器一样说话”的语言。

一点小小的微光，如果能照亮更多的人，就够了。
